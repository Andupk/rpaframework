:Call Stored Procedure:
  :Arguments: spName, spParams=None, sansTran=False

  .. raw:: html

     <p>Uses the inputs of <span class="name">spName</span> and 'spParams' to call a stored procedure. Set optional input <span class="name">sansTran</span> to True to run command without an explicit transaction commit or rollback.</p>
     <p>spName should be the stored procedure name itself spParams [Optional] should be a List of the parameters being sent in.  The list can be one or multiple items.</p>
     <p>The return from this keyword will always be a list.</p>
     <p>Example:</p>
     <table border="1">
     <tr>
     <td>@{ParamList} =</td>
     <td>Create List</td>
     <td>FirstParam</td>
     <td>SecondParam</td>
     <td>ThirdParam</td>
     </tr>
     <tr>
     <td>@{QueryResults} =</td>
     <td>Call Stored Procedure</td>
     <td>DBName.SchemaName.StoredProcName</td>
     <td>List of Parameters</td>
     <td></td>
     </tr>
     </table>
     <p>Example:</p>
     <table border="1">
     <tr>
     <td>@{ParamList} =</td>
     <td>Create List</td>
     <td>Testing</td>
     <td>LastName</td>
     </tr>
     <tr>
     <td>Set Test Variable</td>
     <td>${SPName} =</td>
     <td>DBTest.DBSchema.MyStoredProc</td>
     <td></td>
     </tr>
     <tr>
     <td>@{QueryResults} =</td>
     <td>Call Stored Procedure</td>
     <td>${SPName}</td>
     <td>${ParamList}</td>
     </tr>
     <tr>
     <td>Log List</td>
     <td>@{QueryResults}</td>
     <td></td>
     <td></td>
     </tr>
     </table>
     <p>Using optional <span class="name">sansTran</span> to run command without an explicit transaction commit or rollback:</p>
     <table border="1">
     <tr>
     <td>@{QueryResults} =</td>
     <td>Call Stored Procedure</td>
     <td>DBName.SchemaName.StoredProcName</td>
     <td>List of Parameters</td>
     <td>True</td>
     </tr>
     </table>

:Check If Exists In Database:
  :Arguments: selectStatement, sansTran=False

  .. raw:: html

     <p>Check if any row would be returned by given the input <span class="name">selectStatement</span>. If there are no results, then this will throw an AssertionError. Set optional input <span class="name">sansTran</span> to True to run command without an explicit transaction commit or rollback.</p>
     <p>For example, given we have a table <span class="name">person</span> with the following data:</p>
     <table border="1">
     <tr>
     <td>id</td>
     <td>first_name</td>
     <td>last_name</td>
     </tr>
     <tr>
     <td>1</td>
     <td>Franz Allan</td>
     <td>See</td>
     </tr>
     </table>
     <p>When you have the following assertions in your robot</p>
     <table border="1">
     <tr>
     <td>Check If Exists In Database</td>
     <td>SELECT id FROM person WHERE first_name = 'Franz Allan'</td>
     </tr>
     <tr>
     <td>Check If Exists In Database</td>
     <td>SELECT id FROM person WHERE first_name = 'John'</td>
     </tr>
     </table>
     <p>Then you will get the following:</p>
     <table border="1">
     <tr>
     <td>Check If Exists In Database</td>
     <td>SELECT id FROM person WHERE first_name = 'Franz Allan'</td>
     <td># PASS</td>
     </tr>
     <tr>
     <td>Check If Exists In Database</td>
     <td>SELECT id FROM person WHERE first_name = 'John'</td>
     <td># FAIL</td>
     </tr>
     </table>
     <p>Using optional <span class="name">sansTran</span> to run command without an explicit transaction commit or rollback:</p>
     <table border="1">
     <tr>
     <td>Check If Exists In Database</td>
     <td>SELECT id FROM person WHERE first_name = 'John'</td>
     <td>True</td>
     </tr>
     </table>

:Connect To Database:
  :Arguments: dbapiModuleName=None, dbName=None, dbUsername=None, dbPassword=None, dbHost=None, dbPort=None, dbCharset=None, dbConfigFile=./resources/db.cfg

  .. raw:: html

     <p>Loads the DB API 2.0 module given <span class="name">dbapiModuleName</span> then uses it to connect to the database using <span class="name">dbName</span>, <span class="name">dbUsername</span>, and <span class="name">dbPassword</span>.</p>
     <p>Optionally, you can specify a <span class="name">dbConfigFile</span> wherein it will load the default property values for <span class="name">dbapiModuleName</span>, <span class="name">dbName</span> <span class="name">dbUsername</span> and <span class="name">dbPassword</span> (note: specifying <span class="name">dbapiModuleName</span>, <span class="name">dbName</span> <span class="name">dbUsername</span> or <span class="name">dbPassword</span> directly will override the properties of the same key in <span class="name">dbConfigFile</span>). If no <span class="name">dbConfigFile</span> is specified, it defaults to <span class="name">./resources/db.cfg</span>.</p>
     <p>The <span class="name">dbConfigFile</span> is useful if you don't want to check into your SCM your database credentials.</p>

     dbapiModuleName=pymysqlforexample
     dbName=yourdbname
     dbUsername=yourusername
     dbPassword=yourpassword
     dbHost=yourhost

:Execute Sql Script:
  :Arguments: sqlScriptFileName, sansTran=False

  .. raw:: html

     <p>Executes the content of the <span class="name">sqlScriptFileName</span> as SQL commands. Useful for setting the database to a known state before running your tests, or clearing out your test data after running each a test. Set optional input <span class="name">sansTran</span> to True to run command without an explicit transaction commit or rollback.</p>
     <p>Sample usage :</p>
     <table border="1">
     <tr>
     <td>Execute Sql Script</td>
     <td>${EXECDIR}${/}resources${/}DDL-setup.sql</td>
     </tr>
     <tr>
     <td>Execute Sql Script</td>
     <td>${EXECDIR}${/}resources${/}DML-setup.sql</td>
     </tr>
     <tr>
     <td>#interesting stuff here</td>
     <td></td>
     </tr>
     <tr>
     <td>Execute Sql Script</td>
     <td>${EXECDIR}${/}resources${/}DML-teardown.sql</td>
     </tr>
     <tr>
     <td>Execute Sql Script</td>
     <td>${EXECDIR}${/}resources${/}DDL-teardown.sql</td>
     </tr>
     </table>
     <p>SQL commands are expected to be delimited by a semi-colon (';').</p>
     <p>For example: DELETE FROM person_employee_table; DELETE FROM person_table; DELETE FROM employee_table;</p>
     <p>Also, the last SQL command can optionally omit its trailing semi-colon.</p>
     <p>For example: DELETE FROM person_employee_table; DELETE FROM person_table; DELETE FROM employee_table</p>
     <p>Given this, that means you can create spread your SQL commands in several lines.</p>
     <p>For example: DELETE FROM person_employee_table; DELETE FROM person_table; DELETE FROM employee_table</p>
     <p>However, lines that starts with a number sign (<span class="name">#</span>) are treated as a commented line. Thus, none of the contents of that line will be executed.</p>
     <p>For example: # Delete the bridging table first... DELETE FROM person_employee_table; # ...and then the bridged tables. DELETE FROM person_table; DELETE FROM employee_table</p>
     <p>Using optional <span class="name">sansTran</span> to run command without an explicit transaction commit or rollback:</p>
     <table border="1">
     <tr>
     <td>Execute Sql Script</td>
     <td>${EXECDIR}${/}resources${/}DDL-setup.sql</td>
     <td>True</td>
     </tr>
     </table>

:Query:
  :Arguments: selectStatement, sansTran=False, returnAsDict=False

  .. raw:: html

     <p>Uses the input <span class="name">selectStatement</span> to query for the values that will be returned as a list of tuples. Set optional input <span class="name">sansTran</span> to True to run command without an explicit transaction commit or rollback. Set optional input <span class="name">returnAsDict</span> to True to return values as a list of dictionaries.</p>
     <p>Tip: Unless you want to log all column values of the specified rows, try specifying the column names in your select statements as much as possible to prevent any unnecessary surprises with schema changes and to easily see what your [] indexing is trying to retrieve (i.e. instead of <span class="name">"select * from my_table"</span>, try <span class="name">"select id, col_1, col_2 from my_table"</span>).</p>
     <p>For example, given we have a table <span class="name">person</span> with the following data:</p>
     <table border="1">
     <tr>
     <td>id</td>
     <td>first_name</td>
     <td>last_name</td>
     </tr>
     <tr>
     <td>1</td>
     <td>Franz Allan</td>
     <td>See</td>
     </tr>
     </table>
     <p>When you do the following:</p>
     <table border="1">
     <tr>
     <td>@{queryResults}</td>
     <td>Query</td>
     <td>SELECT * FROM person</td>
     </tr>
     <tr>
     <td>Log Many</td>
     <td>@{queryResults}</td>
     <td></td>
     </tr>
     </table>
     <p>You will get the following: [1, 'Franz Allan', 'See']</p>
     <p>Also, you can do something like this:</p>
     <table border="1">
     <tr>
     <td>${queryResults}</td>
     <td>Query</td>
     <td>SELECT first_name, last_name FROM person</td>
     </tr>
     <tr>
     <td>Log</td>
     <td>${queryResults[0][1]}, ${queryResults[0][0]}</td>
     <td></td>
     </tr>
     </table>
     <p>And get the following See, Franz Allan</p>
     <p>Using optional <span class="name">sansTran</span> to run command without an explicit transaction commit or rollback:</p>
     <table border="1">
     <tr>
     <td>@{queryResults}</td>
     <td>Query</td>

     <td>True</td>

:Set Auto Commit:
  :Arguments: autoCommit=True

  .. raw:: html

     <p>Turn the autocommit on the database connection ON or OFF.</p>
     <p>The default behaviour on a newly created database connection is to automatically start a transaction, which means that database actions that won't work if there is an active transaction will fail. Common examples of these actions are creating or deleting a database or database snapshot. By turning on auto commit on the database connection these actions can be performed.</p>
     <p>Example:</p>
     <pre>
     # Default behaviour, sets auto commit to true
     Set Auto Commit
     # Explicitly set the desired state
     Set Auto Commit | False
     </pre>
